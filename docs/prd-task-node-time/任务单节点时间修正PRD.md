## 功能概述

在任务单管理中新增「修改时间」功能，支持后台人员对任务单的四个关键节点时间进行人工修改，包括

1. 装到达时间；

2. 装离开时间；

3. 卸到达时间；

4. 卸离开时间

后台人员可在任务单列表中点击「修改时间」，打开弹窗后直接修改对应节点时间，并选择修改原因。系统保存修改后的时间，同时记录修改人、修改时间、修改原因、修改前时间和修改后时间，方便后续追溯

本功能主要用于解决司机未操作、司机操作动作错误、后台补录错误等原因导致的节点时间不准确问题

## 需求背景

需求背景：

当前任务单管理列表中展示了任务执行过程中的四个关键时间节点：

| 节点时间 | 说明 |
| --- | --- |
| 装到达时间 | 车辆到达装货点的时间 |
| 装离开时间 | 车辆离开装货点的时间 |
| 卸到达时间 | 车辆到达卸货点的时间 |
| 卸离开时间 | 车辆离开卸货点的时间 |

这四个时间用于记录任务单实际执行过程，是判断任务进度、任务完成情况、装卸耗时、运输耗时、是否超时的重要依据。

当前任务单中的装到达时间、装离开时间、卸到达时间、卸离开时间，是记录任务执行过程的重要节点时间，影响任务进度、运输耗时、装卸耗时、是否超时等业务判断。

但在实际业务中，存在司机未操作、司机操作错误、后台人员补录错误等情况，导致系统记录时间与实际业务发生时间不一致。

为保证任务单节点时间准确，后台需要支持对这四个时间进行快速修改，并记录修改人、修改时间、修改原因、修改前后内容，便于后续追溯

需求目的：

1.通过新增「修改时间」功能，支持后台人员快速修正任务单中的装到达时间、装离开时间、卸到达时间、卸离开时间，解决因司机未操作、司机操作错误、后台补录错误导致的节点时间不准确问题。

2.修改后系统需自动保存修改记录，并同步修改关联功能页面参数字段和重新计算车辆里程保证任务数据准确、可追溯

需求分析：

1. 当前任务单节点时间主要依赖司机操作或后台补录，但这两类数据来源都可能出现问题

2. 司机在实际运输过程中，可能由于忙碌、忘记、不会操作、网络问题、操作习惯差等原因，没有按照业务节点及时操作，导致节点时间不准确

3. 后台人员在补录单据或根据司机反馈修正时间时，也可能因为信息不完整、沟通有误、手工录入错误等原因，造成系统时间仍然不准确

4. 四个节点时间不仅是展示字段，还会影响运单补录效率，车辆里程、业务判断、运营统计分析和对账，因此必须提供可控的修正能力

## 功能边界定义

本次功能只做简单修改，不做复杂流程，功能清单如下：

| 功能 | 说明 |
| --- | --- |
| 修改四个节点时间 | 核心功能 |
| 状态校验 | 仅运输中、已完成允许修改（其他状态不显示） |
| 补录空缺节点时间 | 原时间为空时允许填写 |
| 备注说明 | 必填 |
| 修改日志列表 | 后台自动记录在当前页面生成数据 |
| 修改后重新计算里程 | 按照原有计算逻辑 |
| 关联字段参数修改 | 保证其他功能页面参数取值相同 |
| 功能权限配置 | 将该功能写入操作配置树中 |

如果操作栏空间不足，可放到「更多」中：更多-修改时间

| --- | --- | --- |
| 运输中 | 正常可点击 | 允许修改，保存时二次确认 |
| 已完成 | 正常可点击 | 允许修改，保存时二次确认 |
| 其他状态 | 不展示 |  |

### 弹窗设计

弹窗名称：修改节点时间

#### 弹窗结构

弹窗分为3部分：任务基础信息、节点时间修改、修改原因

#### 任务基础信息

基础信息只做只读展示，用于确认当前修改的任务单

| 字段 | 是否展示 | 是否可编辑 | 说明 |
| --- | --- | --- | --- |
| 任务单号 | 是 | 否 | 当前任务单编号 |
| 车牌号 | 是 | 否 | 当前执行车辆 |
| 司机 | 是 | 否 | 当前司机 |
| 任务状态 | 是 | 否 | 当前任务状态 |

#### 时间节点修改

| 字段 | 类型 | 是否可编辑 | 说明 |
| --- | --- | --- | --- |
| 装到达时间 | 日期时间选择器 | 是 | 可修改或补录 |
| 装离开时间 | 日期时间选择器 | 是 | 可修改或补录 |
| 卸到达时间 | 日期时间选择器 | 是 | 可修改或补录 |
| 卸离开时间 | 日期时间选择器 | 是 | 可修改或补录 |

#### 时间字段交互规则

1. 打开弹窗时，系统默认带出当前已有时间；

2. 如果原时间为空，则对应字段为空，允许后台补录；

3. 用户可只修改其中一个时间；

4. 用户也可同时修改多个时间；

5. 不单独展示「原时间 / 新时间」两列，页面只展示当前可编辑时间；

6. 保存时由系统自动记录修改前值和修改后值；

7. 时间精度建议到秒，格式为：YYYY-MM-DD HH:mm:ss

| 备注说明 | 文本框 | 否 | 必填（字数限制与之前逻辑一样） |

### 状态校验规则

| 任务状态 | 是否允许修改 | 规则说明 |
| --- | --- | --- |
| 待调度 | 不允许 | 任务尚未派车，司机和车辆可能未确定，不应维护实际节点时间 |
| 待运输 | 不允许 | 任务尚未开始执行，不应产生实际装卸节点时间 |
| 运输中 | 允许 | 司机已开始执行任务，允许修正或补录已发生的节点动作时间 |
| 已完成 | 允许 | 任务已完成，允许修正历史节点时间，但可能影响耗时、超时和统计 |

### 时间校验规则

#### 时间顺序校验

保存时，系统需要校验四个节点时间的先后顺序

完整时间顺序为：装到达时间 ≤ 装离开时间 ≤ 卸到达时间 ≤ 卸离开时间

如果不满足，提示：节点时间顺序不正确，请检查后重新保存

#### 部分时间为空时的校验

| 已有时间 | 校验逻辑 |
| --- | --- |
| 只有装到达时间 | 不做前后顺序校验 |
| 有装到达、装离开 | 校验装到达时间 ≤ 装离开时间 |
| 有装离开、卸到达 | 校验装离开时间 ≤ 卸到达时间 |
| 有卸到达、卸离开 | 校验卸到达时间 ≤ 卸离开时间 |
| 四个时间均有值 | 校验完整链路顺序 |

#### 未来时间校验

四个节点时间均不能大于当前系统时间

若用户选择未来时间，提示：节点时间不能大于当前时间。

#### 无变化校验

如果四个节点时间均未发生变化，用户点击保存时提示：当前时间未发生变化，无需保存

### 保存后处理逻辑

用户保存成功后，系统需要执行以下处理：

| 处理事项 | 说明 |
| --- | --- |
| 更新节点时间（pc+小程序） | 更新任务单四个节点时间 |
| 记录修改日志 | 记录修改人、修改时间、修改原因、修改前后值 |
| 重新计算车辆里程 | 按照原有逻辑重新计算车辆里程 |
| 同步相关功能页面参数取值 | 修改时间关联字段 ![文档图片](assets/image4.png) |
| 任务单详情添加操作记录 | 【修改装到达时间】修改前值：X 修改后值：N （平台；操作人：xxx；操作时间：YYYY-MM-DD HH:mm:ss） |

### 修改日志

#### 日志记录要求

系统保存节点时间修改时，必须记录修改日志，不允许无痕覆盖。

#### 修改日志列表字段

| 字段 | 说明 |
| --- | --- |
| 任务单号 | 被修改的任务单 |
| 修改人 | 当前操作人 |
| 修改时间 | 用户点击保存成功的时间 |
| 备注说明 | 用户填写的备注 |
| 修改字段 | 被修改的节点时间字段 |
| 修改前值 | 修改前时间 |
| 修改后值 | 修改后时间 |

#### 筛选条件：

修改时间、任务单号、修改人
